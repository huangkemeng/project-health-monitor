## 1. 功能概述

### 1.1 产品愿景

产品上线初期，用户在使用过程中难免会遇到各类问题（如功能异常、界面显示错误、性能问题、功能建议等）。本功能旨在为**项目健康监控系统**建立一个畅通的用户反馈渠道，让用户能够方便快捷地提交问题和建议，帮助开发团队及时发现、追踪和修复问题，持续提升产品质量和用户体验。

### 1.2 目标用户

| 用户角色 | 描述 | 核心需求 |
|---------|------|---------|
| **系统用户** | 使用项目健康监控系统的所有注册用户 | 提交使用中遇到的问题、功能建议、改进意见 |
| **未登录访客** | 尚未注册或登录的潜在用户 | 提交登录页、公开页面等遇到的问题 |
| **系统管理员** | 负责系统运维和需求处理的技术负责人 | 查看反馈列表、管理反馈状态、跟进处理进度 |
| **开发人员** | 负责系统开发和Bug修复的工程师 | 获取详细反馈信息、复现问题、标记处理结果 |

### 1.3 业务价值

| 价值维度 | 价值描述 |
|---------|---------|
| **快速响应** | 用户发现问题后可直接在系统内提交，降低反馈门槛，缩短问题发现到修复的周期 |
| **闭环管理** | 从反馈提交到处理完成的完整生命周期管理，确保每个反馈都有着落 |
| **质量提升** | 系统化收集用户反馈，帮助开发团队持续优化产品，提升稳定性 |
| **用户信任** | 用户看到自己的反馈被重视和处理，增强对产品的信任感和参与感 |

### 2.2 反馈提交功能

#### 2.2.1 功能描述

用户可在系统任意页面通过固定的入口（如右下角浮动按钮）打开反馈提交表单，填写问题描述并提交。系统自动捕获当前页面信息（URL、浏览器信息等），帮助开发人员快速定位问题。

#### 2.2.2 交互入口

| 入口位置 | 触发方式 | 说明 |
|---------|---------|------|
| 页面右下角浮动按钮 | 点击 | 始终悬浮显示，图标为"气泡/留言"样式 |
| 用户头像下拉菜单 | 点击 | 提供"问题反馈"菜单项 |
| 设置页面 | 点击 | 在设置页面提供反馈入口 |

#### 2.2.3 反馈表单字段

| 字段名 | 类型 | 必填 | 校验规则 | 说明 |
|-------|------|:----:|---------|------|
| 反馈类型 | 单选 | 是 | `bug` / `feature_request` / `other` | 问题类型选择 |
| 标题 | 文本 | 是 | 5-100字符 | 问题简要描述 |
| 详细描述 | 多行文本 | 是 | 10-2000字符 | 详细描述问题或建议 |
| 复现步骤 | 多行文本 | 否(仅Bug类型时建议) | 0-2000字符 | Bug复现步骤 |
| 期望行为 | 多行文本 | 否 | 0-1000字符 | 期望的正确行为 |
| 实际行为 | 多行文本 | 否 | 0-1000字符 | 实际出现的错误行为 |
| 附件 | 文件上传 | 否 | jpg/png/gif/webp/log/txt，单文件≤5MB，最多5个 | 截图或日志文件 |
| 联系方式 | 文本 | 否 | 邮箱或手机号 | 如需进一步沟通时使用 |

#### 2.2.4 自动捕获信息

| 信息项 | 捕获方式 | 说明 |
|-------|---------|------|
| 当前页面URL | 自动 | 用户提交时的页面URL |
| 用户浏览器信息 | 自动 | User-Agent字符串 |
| 浏览器语言 | 自动 | navigator.language |
| 屏幕分辨率 | 自动 | screen.width × screen.height |
| 操作系统 | 自动 | 从User-Agent解析 |
| 当前用户邮箱 | 自动（已登录） | 从认证信息获取 |
| 系统版本号 | 自动 | 从构建信息获取 |
| 请求时间 | 自动 | 服务端记录提交时间 |

**注意**：自动捕获信息应在表单底部以展开/收起的隐私说明形式展示，让用户知晓我们收集了哪些信息。

#### 2.2.5 上传附件规则

1. **文件类型限制**：仅支持 `jpg`、`png`、`gif`、`webp`、`log`、`txt` 格式
2. **单文件大小**：不超过 5MB
3. **文件数量**：最多 5 个附件
4. **自动压缩**：图片类附件超过 2MB 时自动压缩到 2MB 以内
5. **上传方式**：用户选择文件后立即上传到服务端，表单提交时只保存文件ID列表
6. **上传进度**：显示文件上传进度条，支持取消上传
7. **错误处理**：
   - 文件类型不支持时提示："不支持的文件格式，支持的格式：jpg、png、gif、webp、log、txt"
   - 文件过大时提示："文件大小不能超过5MB"
   - 上传失败时提示："文件上传失败，请重试"

#### 2.2.6 提交后交互

1. 提交成功后，页面显示成功提示，并提供反馈编号（如 `FB-20260505-0001`）
2. 用户可选择"查看我的反馈"跳转到该反馈的详情页
3. 自动发送确认通知（站内通知）

#### 2.2.7 业务规则

| 规则编号 | 规则描述 |
|:-------:|---------|
| R001 | 已登录用户提交时自动关联用户ID |
| R002 | 未登录用户可填写邮箱后提交（需验证邮箱验证码，V2版本支持） |
| R003 | 同一用户短时间内（5分钟内）最多提交5次反馈，防刷机制 |
| R004 | 提交成功后自动创建一条追踪记录 |
| R005 | 自动捕获的系统信息需要在隐私声明中说明用途 |

### 2.3 反馈列表功能

#### 2.3.1 功能描述

用户可在"我的反馈"页面查看自己提交的所有反馈记录，了解每条反馈的当前处理状态。

#### 2.3.2 列表展示

| 展示字段 | 说明 |
|---------|------|
| 反馈编号 | 唯一标识，如 `FB-20260505-0001` |
| 标题 | 反馈标题 |
| 反馈类型 | Bug/建议/其他，以标签形式展示 |
| 提交时间 | 格式化时间显示 |
| 当前状态 | 以状态标签形式展示，不同状态不同颜色 |
| 最近更新 | 最后一次更新时间 |

**列表布局**：
- 默认按提交时间倒序排列
- 支持分页，每页20条
- 空状态时显示引导："暂无反馈记录，遇到问题？[提交反馈]"

#### 2.3.3 筛选与搜索

| 筛选维度 | 筛选选项 | 默认值 |
|---------|---------|:-----:|
| 状态 | 全部 / 待处理 / 处理中 / 已修复 / 已关闭 / 重复反馈 | 全部 |
| 类型 | 全部 / Bug / 功能建议 / 其他 | 全部 |
| 时间范围 | 最近7天 / 最近30天 / 最近90天 / 自定义 | 最近30天 |

**关键词搜索**：
- 支持按标题和反馈编号进行模糊搜索
- 搜索输入框带防抖（300ms）

#### 2.3.4 交互功能

1. 点击反馈条目进入反馈详情页
2. 鼠标悬停时显示简短预览（前50字描述）
3. 支持批量操作（选择多个反馈进行批量关闭）

### 2.4 反馈详情功能

#### 2.4.1 功能描述

展示单条反馈的完整信息和处理进度，支持用户与管理员之间的回复沟通。

#### 2.4.2 页面布局

**顶部区域 - 反馈概要**：
| 字段 | 说明 |
|-----|------|
| 反馈编号 | 显眼展示 |
| 标题 | 大字号标题 |
| 反馈类型 | 类型标签 |
| 当前状态 | 状态标签 |
| 提交时间 | 格式化时间 |
| 提交人 | 用户名/邮箱 |

**中部区域 - 反馈内容**：
| 区块 | 内容 |
|-----|------|
| 详细描述 | 用户的详细问题描述 |
| 复现步骤 | 如为Bug类型且填写了复现步骤 |
| 期望行为 | 如填写 |
| 实际行为 | 如填写 |
| 附件列表 | 缩略图展示，点击放大查看/下载 |
| 系统信息 | 自动捕获的页面URL、浏览器、操作系统等（以折叠面板展示） |

**底部区域 - 处理进度与回复**（按时间倒序排列）：

```
[系统消息] 2026-05-05 14:30 - 反馈已提交，状态：待处理
[管理员回复] 2026-05-05 15:00 - 已确认问题，正在排查...
[系统消息] 2026-05-05 16:30 - 状态变更为：处理中
[用户回复]   2026-05-05 17:00 - 好的，请尽快修复，谢谢
[管理员回复] 2026-05-05 18:00 - 已修复，将在下次部署时上线，请验证
[系统消息] 2026-05-06 09:00 - 状态变更为：已修复
[用户回复]   2026-05-06 10:00 - 已验证，问题已解决，感谢！
[系统消息] 2026-05-06 10:00 - 状态变更为：已关闭
```

**底部输入框**：
- 用户可输入文字回复
- 支持上传附件（同提交规则）
- 点击"发送"按钮提交回复

#### 2.4.3 业务规则

| 规则编号 | 规则描述 |
|:-------:|---------|
| R006 | 只有反馈提交人和管理员可以查看/回复该反馈 |
| R007 | 反馈关闭后，用户可重新开启（7天内） |
| R008 | 每次状态变更和回复都生成一条时间线记录 |
| R009 | 管理员回复时，回复者显示为"管理员"或真实姓名 |

### 2.5 状态管理功能

#### 2.5.1 状态定义

| 状态 | 含义 | 颜色标识 | 可操作角色 |
|:----:|------|:-------:|-----------|
| **待处理** | 新提交的反馈，尚未处理 | 🔴 红色 | 管理员可接收处理 |
| **处理中** | 已确认问题，正在排查/修复 | 🟡 黄色 | 管理员进行处理，用户可查看 |
| **已修复** | 问题已修复，待用户验证 | 🟢 绿色 | 用户可验证并关闭 |
| **已关闭** | 用户确认已解决或主动关闭 | ⚪ 灰色 | 用户和管理员可关闭 |
| **重复反馈** | 与已有反馈重复，关联到原反馈 | 🔵 蓝色 | 管理员可标记 |

#### 2.5.2 状态流转图

```
                          ┌──────────────┐
                          │   待处理      │
                          └──────┬───────┘
                                 │
                    ┌────────────┼────────────┐
                    │            │            │
                    ▼            ▼            ▼
             ┌──────────┐ ┌──────────┐ ┌──────────┐
             │ 处理中    │ │ 重复反馈 │ │  已关闭  │
             └─────┬────┘ └─────┬────┘ └──────────┘
                   │            │          ▲
                   ▼            │          │
             ┌──────────┐       ├──────────┘
             │ 已修复    │       │  (关联原反馈后关闭)
             └─────┬────┘       │
                   │            │
                   ▼            │
             ┌──────────┐       │
             │ 已关闭    │◄──────┘
             └──────────┘
                    
        状态说明:
        ──────────────────────────────────────────
        • 待处理  → 处理中  : 管理员确认并开始处理
        • 待处理  → 重复反馈: 管理员标记为重复
        • 待处理  → 已关闭  : 管理员判定非问题
        • 处理中  → 已修复  : 开发完成修复
        • 已修复  → 已关闭  : 用户验证通过后关闭
        • 已修复  → 处理中  : 用户验证未通过
        • 任何状态 → 已关闭  : 用户或管理员主动关闭
        • 已关闭  → 待处理  : 用户7天内重新开启
        ──────────────────────────────────────────
```

#### 2.5.3 状态变更规则

| 规则编号 | 规则描述 |
|:-------:|---------|
| R010 | 状态变更时，必须有备注说明变更原因 |
| R011 | 标记为"重复反馈"时，必须指定关联的原始反馈编号 |
| R012 | 标记为"已修复"时，建议备注修复版本号或PR链接 |
| R013 | 状态变更后，系统自动通知反馈提交人 |

### 4.2 表结构设计

#### 4.2.1 feedbacks 表

用户反馈主表，存储每条反馈的核心信息。

```sql
CREATE TABLE feedbacks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    feedback_no VARCHAR(20) NOT NULL UNIQUE,          -- 反馈编号，如 FB-20260505-0001
    user_id UUID REFERENCES users(id) ON DELETE SET NULL,  -- 提交人ID（未登录时为NULL）
    title VARCHAR(100) NOT NULL,                       -- 反馈标题（5-100字符）
    type VARCHAR(20) NOT NULL CHECK (type IN ('bug', 'feature_request', 'other')),  -- 反馈类型
    description TEXT NOT NULL,                         -- 详细描述（10-2000字符）
    steps TEXT,                                        -- 复现步骤（JSON数组或纯文本）
    expected_behavior TEXT,                            -- 期望行为
    actual_behavior TEXT,                              -- 实际行为
    status VARCHAR(20) NOT NULL DEFAULT 'pending' 
        CHECK (status IN ('pending', 'processing', 'resolved', 'closed', 'duplicate')),  -- 当前状态
    assigned_to UUID REFERENCES users(id) ON DELETE SET NULL,  -- 分配处理人
    duplicated_to UUID REFERENCES feedbacks(id) ON DELETE SET NULL,  -- 重复关联的反馈ID
    contact VARCHAR(100),                              -- 联系方式（邮箱/手机）
    
    -- 自动捕获信息
    page_url TEXT,                                     -- 提交时的页面URL
    user_agent TEXT,                                   -- 浏览器User-Agent
    os_info VARCHAR(100),                              -- 操作系统信息
    browser_info VARCHAR(100),                         -- 浏览器信息
    screen_resolution VARCHAR(20),                     -- 屏幕分辨率
    app_version VARCHAR(20),                           -- 系统版本号
    
    -- 时间信息
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    closed_at TIMESTAMP WITH TIME ZONE,                -- 关闭时间
    
    -- 索引
    CONSTRAINT fk_assigned_to FOREIGN KEY (assigned_to) REFERENCES users(id),
    CONSTRAINT fk_duplicated_to FOREIGN KEY (duplicated_to) REFERENCES feedbacks(id)
);

-- 索引
CREATE INDEX idx_feedbacks_user_id ON feedbacks(user_id);
CREATE INDEX idx_feedbacks_status ON feedbacks(status);
CREATE INDEX idx_feedbacks_type ON feedbacks(type);
CREATE INDEX idx_feedbacks_created_at ON feedbacks(created_at DESC);
CREATE INDEX idx_feedbacks_assigned_to ON feedbacks(assigned_to);
CREATE INDEX idx_feedbacks_feedback_no ON feedbacks(feedback_no);
CREATE INDEX idx_feedbacks_status_created ON feedbacks(status, created_at DESC);
```

| user_id | UUID | 否 | 提交人ID，未登录时为NULL（V2支持） |

| steps | TEXT | 否 | 复现步骤，JSON数组格式或纯文本 |
| expected_behavior | TEXT | 否 | 期望行为 |
| actual_behavior | TEXT | 否 | 实际行为 |

| created_at | TIMESTAMP | 是 | 上传时间 |

#### 4.2.4 feedback_timeline 表

反馈时间线表，记录反馈的状态变更历史。

```sql
CREATE TABLE feedback_timeline (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    feedback_id UUID NOT NULL REFERENCES feedbacks(id) ON DELETE CASCADE,
    action_type VARCHAR(30) NOT NULL 
        CHECK (action_type IN ('created', 'status_changed', 'replied', 'assigned')),
    from_status VARCHAR(20),                           -- 变更前状态（status_changed时必填）
    to_status VARCHAR(20),                             -- 变更后状态（status_changed时必填）
    note TEXT,                                         -- 变更备注
    operator_id UUID REFERENCES users(id) ON DELETE SET NULL,  -- 操作人ID
    operator_name VARCHAR(100),                       -- 操作人显示名称
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    
    CONSTRAINT fk_timeline_feedback FOREIGN KEY (feedback_id) REFERENCES feedbacks(id),
    CONSTRAINT fk_timeline_operator FOREIGN KEY (operator_id) REFERENCES users(id)
);

CREATE INDEX idx_timeline_feedback_id ON feedback_timeline(feedback_id);
CREATE INDEX idx_timeline_created_at ON feedback_timeline(feedback_id, created_at DESC);
```

| 字段名 | 类型 | 必填 | 说明 |
|-------|------|:----:|------|
| id | UUID | 是 | 主键 |
| feedback_id | UUID | 是 | 关联的反馈ID |
| action_type | VARCHAR(30) | 是 | 操作类型：created（创建）/ status_changed（状态变更）/ replied（回复）/ assigned（分配） |
| from_status | VARCHAR(20) | 否 | 变更前状态（action_type=status_changed时必填） |
| to_status | VARCHAR(20) | 否 | 变更后状态（action_type=status_changed时必填） |
| note | TEXT | 否 | 变更备注说明 |
| operator_id | UUID | 否 | 操作人ID |
| operator_name | VARCHAR(100) | 否 | 操作人显示名称 |
| created_at | TIMESTAMP | 是 | 操作时间 |

### 4.3 数据库规范建议

#### 4.3.1 存储策略

| 项目 | 建议 |
|-----|------|
| **附件存储** | 使用对象存储（如AWS S3、阿里云OSS），本地仅保存路径引用 |
| **文件目录** | 按日期分目录：`feedback/{yyyy}/{mm}/{dd}/{uuid}.{ext}` |
| **清理策略** | 附件保留至少180天，超期可归档至冷存储 |
| **数据归档** | 已关闭超过90天的反馈可归档至历史表，减少主表数据量 |

#### 4.3.2 反馈编号生成策略

```
格式: FB-{yyyyMMdd}-{4位序号}

示例: FB-20260505-0001

生成规则:
1. 使用数据库序列或 Redis INCR 命令生成每日自增序号
2. 序号每日重置，每天从 0001 开始
3. 使用应用层生成避免数据库顺序锁
4. 高并发场景可使用雪花算法生成全局唯一ID作为feedback_no
```

#### 4.3.3 查询优化建议

| 查询场景 | 优化建议 |
|---------|---------|
| 我的反馈列表 | 在 `feedbacks` 表的 `(user_id, created_at DESC)` 上建立复合索引 |
| 管理后台列表 | 在 `feedbacks` 表的 `(status, created_at DESC)` 上建立复合索引 |
| 统计概览 | 使用物化视图或定时汇总表缓存统计数据 |
| 搜索 | 如数据量大，可引入 Elasticsearch 做全文搜索 |

## 5. 非功能需求

### 5.1 性能要求

| 指标 | 要求 |
|-----|------|
| 反馈提交响应时间 | ≤ 1s（不含附件上传时间） |
| 附件上传响应时间 | ≤ 3s（5MB文件，正常网络条件） |
| 反馈列表查询 | ≤ 500ms |
| 反馈详情查询 | ≤ 300ms |
| 并发支持 | 支持至少50人同时提交反馈 |

### 5.2 安全要求

| 安全项 | 要求 |
|-------|------|
| 文件上传 | 严格校验文件类型，防止上传可执行文件（exe、sh、php等） |
| XSS防护 | 反馈内容中的HTML标签需转义或过滤 |
| 访问控制 | 用户只能查看自己的反馈，管理员可查看全部 |
| 频率限制 | 同一用户5分钟内最多提交5次反馈 |
| 文件扫描 | 上传文件需进行病毒扫描（推荐） |

### 5.3 隐私要求

| 隐私项 | 说明 |
|-------|------|
| 自动捕获信息 | 需在提交表单底部展示隐私说明，告知用户我们会收集哪些信息 |
| 用户联系方式 | 仅用于反馈沟通，不得用于其他用途 |
| 数据保留 | 用户可申请删除自己的反馈数据 |
| 数据导出 | 用户可导出自己的全部反馈数据 |
