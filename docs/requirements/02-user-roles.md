│   │  │   数据隔离    │  │   功能权限    │                │   │
│   │  │  仅访问自己   │  │  完整功能     │                │   │

│                                                              │
└─────────────────────────────────────────────────────────────┘

### 2.3.2 角色详细定义

#### 普通用户 (Standard User)

| 属性 | 描述 |
|-----|------|
| **角色名称** | 普通用户 / Standard User |
| **角色标识** | `ROLE_USER` |
| **角色描述** | 系统的标准用户，通常是开发人员，可以管理自己的监控项和配置 |
| **用户群体** | 后端开发工程师、全栈开发工程师、技术负责人 |
| **角色数量** | 多个（所有注册用户） |

**核心职责**:
1. 配置和管理自己负责的监控项
2. 接收和处理监控告警
3. 查看监控状态和历史数据
4. 配置企业微信Webhook

**数据范围**:
- 只能访问自己创建的监控项
- 只能访问自己配置的Webhook
- 只能查看自己的告警历史

## 2.5 数据隔离规则

### 2.5.1 隔离策略

本系统采用**基于用户ID的数据行级隔离**策略：

```
┌─────────────────────────────────────────────────────────────┐
│                     数据隔离模型                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   用户A (user_id: A)          用户B (user_id: B)             │
│   ┌─────────────────┐         ┌─────────────────┐           │
│   │  监控项A1        │         │  监控项B1        │           │
│   │  监控项A2        │         │  监控项B2        │           │
│   │  监控项A3        │         │  监控项B3        │           │
│   └─────────────────┘         └─────────────────┘           │
│                                                              │
│   数据查询规则: WHERE owner_id = 当前用户ID                   │
│                                                              │
│   隔离级别: 行级隔离（Row-level Security）                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.5.2 隔离实现规则

#### 规则1: 所有业务数据必须包含owner_id字段

```sql
-- 监控表示例
CREATE TABLE monitors (
    id UUID PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    url VARCHAR(500) NOT NULL,
    -- ... 其他字段
    owner_id UUID NOT NULL,  -- 数据隔离关键字段
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (owner_id) REFERENCES users(id)
);

-- 创建索引优化查询
CREATE INDEX idx_monitors_owner ON monitors(owner_id);
```

#### 规则2: 所有查询必须带owner_id过滤

```python
# 正确的查询方式
def get_user_monitors(user_id):
    return db.query(Monitor).filter(Monitor.owner_id == user_id).all()

# 错误的查询方式（会导致数据泄露）
def get_all_monitors():
    return db.query(Monitor).all()  # ❌ 危险操作
```

#### 规则3: 服务端必须验证数据归属

```python
# 更新操作前验证归属
def update_monitor(monitor_id, user_id, data):
    monitor = db.query(Monitor).filter(
        Monitor.id == monitor_id,
        Monitor.owner_id == user_id  # 确保只能操作自己的数据
    ).first()
    
    if not monitor:
        raise PermissionError("无权访问该监控项")
    
    # 执行更新...
```

## 2.6 认证与授权流程

### 2.6.1 认证流程

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│    客户端     │────▶│   登录接口    │────▶│   认证服务    │────▶│   用户数据库  │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
      │                    │                    │                    │
      │ 1.提交用户名密码    │                    │                    │
      │───────────────────▶│                    │                    │
      │                    │ 2.验证凭据          │                    │
      │                    │───────────────────▶│                    │
      │                    │                    │ 3.查询用户信息      │
      │                    │                    │───────────────────▶│
      │                    │                    │◀───────────────────│
      │                    │◀───────────────────│ 4.返回用户信息      │
      │◀───────────────────│ 5.生成JWT Token   │                    │
      │ 6.返回Token        │                    │                    │
      │                    │                    │                    │

后续请求:
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│    客户端     │────▶│   API接口    │────▶│   业务服务    │
└──────────────┘     └──────────────┘     └──────────────┘
      │                    │                    │
      │ 1.请求API (带Token) │                    │
      │───────────────────▶│                    │
      │                    │ 2.验证Token         │
      │                    │ 3.提取user_id       │
      │                    │───────────────────▶│
      │                    │                    │ 4.执行业务逻辑
      │                    │                    │ (带user_id过滤)
      │                    │◀───────────────────│
      │◀───────────────────│ 5.返回数据          │
```

### 2.6.2 授权验证流程

```
┌─────────────────────────────────────────────────────────────┐
│                    请求授权验证流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 接收请求                                                 │
│     └── 从Header提取: Authorization: Bearer <token>          │
│                                                              │
│  2. 验证Token                                                │
│     └── 验证JWT签名和有效期                                   │
│     └── 无效Token → 返回401 Unauthorized                      │
│                                                              │
│  3. 提取用户身份                                              │
│     └── 从Token解析user_id                                   │
│                                                              │
│  4. 权限检查                                                  │
│     └── 检查用户是否有权限访问该API                           │
│     └── 无权限 → 返回403 Forbidden                            │
│                                                              │
│  5. 数据隔离检查                                              │
│     └── 在查询中添加: WHERE owner_id = <user_id>             │
│     └── 资源不存在 → 返回404 Not Found                        │
│                                                              │
│  6. 执行业务逻辑                                              │
│                                                              │
│  7. 返回结果                                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 2.7 安全要求

### 2.7.1 密码安全

| 要求项 | 具体要求 | 实现方式 |
|-------|---------|---------|
| 密码存储 | 禁止明文存储 | 使用bcrypt算法哈希存储 |
| 密码复杂度 | 最小8位，包含字母和数字 | 注册/修改时前端+后端校验 |
| 密码传输 | 传输过程加密 | 全站HTTPS |
| 密码重试 | 防止暴力破解 | 连续5次失败后锁定15分钟 |

### 2.7.2 Token安全

| 要求项 | 具体要求 | 实现方式 |
|-------|---------|---------|
| Token类型 | JWT (JSON Web Token) | 使用HS256算法签名 |
| Token有效期 | 7天 | 设置exp claim |
| Token刷新 | 支持自动刷新 | 使用refresh token机制 |
| Token撤销 | 支持登出失效 | 使用黑名单机制（可选） |

### 2.7.3 会话安全

| 要求项 | 具体要求 | 实现方式 |
|-------|---------|---------|
| 登录状态 | 服务端可控制 | Token有效期控制 |
| 并发登录 | 允许多端登录 | 不限制设备数 |
| 异常检测 | 检测异常登录 | 记录登录IP和时间 |

## 2.8 用户体验设计原则

### 2.8.1 登录体验

| 场景 | 设计原则 |
|-----|---------|
| 首次访问 | 自动跳转登录页，登录后记住跳转目标 |
| Token过期 | 自动刷新Token，刷新失败再跳转登录 |
| 权限不足 | 友好提示"无权访问"，提供返回首页入口 |
| 资源不存在 | 返回404页面，不暴露资源是否存在 |

### 2.8.2 数据可见性

| 场景 | 设计原则 |
|-----|---------|
| 列表页 | 默认只显示自己的数据，无需额外筛选 |
| 详情页 | 访问他人资源时返回404，不提示"无权限" |
| 统计信息 | 大盘只统计自己的监控项数据 |
